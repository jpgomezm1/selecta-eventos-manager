# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Gestor interno de eventos de catering para **Selecta**. Frontend SPA (Vite + React + TypeScript + shadcn-ui + Tailwind) con backend en **Supabase** (Postgres, Auth, Edge Functions, Storage). Originalmente generado en Lovable — cambios hechos aquí se reflejan allá si se hace push.

El working directory raíz tiene un `package-lock.json` vestigial, **el proyecto real está en `selecta-eventos-manager/`** — todos los comandos se corren desde ahí.

## Commands

All commands run from `selecta-eventos-manager/`:

```sh
npm i               # install (npm is used despite bun.lockb being present; package-lock.json is the source of truth)
npm run dev         # vite dev server on port 4000 (strictPort: false — may pick another if occupied)
npm run build       # production build → dist/
npm run build:dev   # build with mode=development (keeps componentTagger for Lovable)
npm run lint        # eslint .
npm run preview     # preview built bundle
```

No test runner is configured. No typecheck script — use `npx tsc --noEmit -p tsconfig.app.json` if needed.

Deployment target is **Netlify** (`netlify.toml` — publish `dist/`). SPA redirects are commented out in netlify.toml; if direct URL navigation 404s in prod, that's why.

## Environment

`.env` (gitignored, present locally) must define:

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`

Client initialized in [src/integrations/supabase/client.ts](src/integrations/supabase/client.ts). Sessions persist in `localStorage`.

The Supabase edge function `generate-recipe` requires `ANTHROPIC_API_KEY` set as a Supabase secret — it is a thin proxy to `api.anthropic.com/v1/messages` used by the Recetario AI features.

## Architecture

### Routing & auth

[src/App.tsx](src/App.tsx) wires every authenticated route through `<ProtectedRoute>`, which gates on `useAuth()` and wraps children in `<AppLayout>`. Public routes: `/auth` and `/compartido/:token` (cotización pública compartida). Legacy `/cotizador/*` paths redirect to `/cotizaciones/*`.

Auth state comes from [src/hooks/useAuth.tsx](src/hooks/useAuth.tsx) — a context subscribing to `supabase.auth.onAuthStateChange`. **There is no `login`/`signOut` in the context**; pages call `supabase.auth` directly. Do not add auth logic to new hooks — extend the existing provider if needed.

### Data layer

All Supabase access goes through the api modules in [src/integrations/supabase/](src/integrations/supabase/) — one file per domain (`apiCotizador.ts`, `apiClientes.ts`, `apiMenaje.ts`, `apiTransporte.ts`, `apiInventario.ts`, `apiCatalogos.ts`, `apiOrdenCompra.ts`, `apiEventoChecklist.ts`, `apiShare.ts`). Components and pages should NOT import `supabase` directly for business queries — add a function to the relevant api module. `types.ts` is auto-generated by Supabase CLI; `client.ts` comment says "do not edit directly".

React Query (`@tanstack/react-query`) is provided at the root. Prefer `useQuery` / `useMutation` calling into the api modules over ad-hoc `useEffect` + setState.

### Domain model (big picture)

The app is organized around 7 business domains. Each has a page (`src/pages/*.tsx`), a component folder (`src/components/<Domain>/`), and an api module:

1. **Personal** — empleados, tarifas, modalidades de cobro, liquidaciones, carga masiva desde Excel.
2. **Cotizaciones** — wizard-based quote builder. A `Cotizacion` has multiple **versiones**, each with multiple **opciones** (A/B/C), each with items (platos, personal, transportes, menaje). Public share tokens live in `apiShare.ts`. State machine managed via SQL updates in `sql_updates_estados_cotizacion.sql`.
3. **Eventos** — an accepted cotización becomes an evento. Panels per-concern (`MenajePanel`, `PersonalPanel`, `TransportePanel`, `OrdenCompraPanel`, `CierreEventoPanel`) + checklist.
4. **Pipeline** — kanban over cotizaciones (stages in `pipeline_columns.sql`).
5. **Bodega / Menaje** — inventario de menaje con reservas y calendario.
6. **Inventario (Insumos)** — ingredientes, movimientos de stock, lectura de facturas (`src/services/invoiceScanner.ts` → Anthropic).
7. **Recetario** — platos + ingredientes + generación AI vía edge function.

Supporting: **Catálogos** (platos/personal/transporte master data), **Clientes**, **Transporte**.

### Cotizador wizard

The quote builder is the most complex flow. State lives in [src/hooks/useCotizadorWizard.ts](src/hooks/useCotizadorWizard.ts) — a single hook holding steps, cliente, lugares, and an array of `OpcionState` (each one an independent `CotizacionItemsState`). Step components in [src/components/Cotizador/](src/components/Cotizador/) are pure UI reading/writing this hook's state. New selectors should follow the existing `*Selector.tsx` pattern.

### Modalidad de cobro for personal

Personal is billed under 6 modalities (`por_hora`, `jornada_9h`, `jornada_10h`, `jornada_hasta_10h`, `jornada_nocturna`, `por_evento`). All payment math goes through [src/lib/calcularPagoPersonal.ts](src/lib/calcularPagoPersonal.ts) — never hand-roll `tarifa * horas` in components. The `jornada_hasta_10h` modality is the only one that uses `tarifa_hora_extra`. See [INSTRUCCIONES_MODALIDAD_COBRO.md](INSTRUCCIONES_MODALIDAD_COBRO.md) for the full table. There are TWO tables with `modalidad_cobro`: `personal` (employees) and `personal_costos_catalogo` (used by cotizador). Changes to billing logic usually touch both.

### Bulk Personal upload

Excel import lives in [src/components/Forms/CargaMasivaPersonal.tsx](src/components/Forms/CargaMasivaPersonal.tsx) backed by [src/lib/mapeoExcelPersonal.ts](src/lib/mapeoExcelPersonal.ts). The mapping tables (rol → rol BD, modalidad → modalidad BD) are the contract with the user's Excel format — see [INSTRUCCIONES_CARGA_MASIVA.md](INSTRUCCIONES_CARGA_MASIVA.md). Adding a new rol or modalidad requires updating both the mapping constant AND the Spanish docs — the user relies on those tables.

### PDF generation

Multiple generators in `src/lib/` (`pdf-generator.ts`, `premium-pdf-generator.ts`, `selecta-premium-pdf.ts`, `orden-compra-pdf.ts`, `orden-menaje-pdf.ts`). They use `jspdf` + `html2canvas`. When adding a new document type, match the existing pattern (function returning a `Blob`/saving directly) rather than introducing a new PDF lib.

### Database migrations

Timestamped migrations live in [supabase/migrations/](supabase/migrations/) but schema changes are also checked in as loose files at project root (`sql_updates_*.sql`, `supabase/migrations/*.sql` without timestamps like `share_tokens.sql`, `pipeline_columns.sql`, `cotizacion_lugares_precio.sql`). These **are not run automatically** — the workflow is: write the SQL, paste into Supabase SQL Editor, run manually. Record the change both as a migration file AND update `src/integrations/supabase/types.ts` (re-generate if using Supabase CLI). When asked to add a column/table, produce the SQL and tell the user to run it.

## Conventions

- Path alias: `@/` → `./src/` (configured in both `vite.config.ts` and `tsconfig.json`).
- UI components are shadcn-ui, installed into [src/components/ui/](src/components/ui/). Base color is `slate`, CSS variables enabled. Add new shadcn components via `npx shadcn@latest add <component>`.
- Toasts: two systems coexist — `@/components/ui/toaster` (shadcn/radix) and `sonner`. Prefer `sonner` for new code; `useToast()` is still used in legacy places.
- Icons: `lucide-react`.
- Dates: `date-fns`.
- Forms: `react-hook-form` + `zod` + `@hookform/resolvers`.
- Styling: Tailwind only. `tailwind.config.ts` defines the Selecta palette — reuse existing tokens (`primary`, `accent`, etc.) instead of raw hex.
- The UI copy is Spanish. Keep user-facing strings and domain identifiers (`cotizacion`, `evento`, `personal`, `menaje`, `recetario`) in Spanish; keep code comments and variable names consistent with what's already in the file (mixed Spanish/English is normal here).
